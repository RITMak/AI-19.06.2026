# План: FSM (Finite State Machine) для чата

## Общая концепция

Каждый чат — конечный автомат с этапами. LLM управляет шагами внутри этапа через маркеры в ответе. Пользователь подтверждает переход между этапами либо нажатием кнопки, либо текстом в чате (LLM сама понимает контекст).

---

## Жизненный цикл чата

```
PLANNING ──→ EXECUTION ──→ DONE
                ↑              │
                └── REOPEN ────┘
```

| Этап | Описание |
|------|---------|
| **PLANNING** | LLM составляет план, уточняет детали. Внутри этапа шаги меняются автоматически. |
| **EXECUTION** | LLM выполняет шаги плана. Внутри этапа шаги меняются автоматически. |
| **DONE** | Задача завершена. Можно продолжить общение или вернуться в EXECUTION. |
| **REOPEN** | Возврат из DONE обратно в EXECUTION (по просьбе пользователя или предложению LLM). |

---

## Поведение системы

### Старт чата
Новый чат автоматически входит в **PLANNING**. Пользователь просто пишет запрос.

### Внутри этапа — автоматическое управление шагами (LLM)
- LLM получает system prompt: `[Stage: EXECUTION, Step: 2/3]`
- LLM сама решает когда шаг выполнен и переключает следующий
- Маркер в ответе: `[STEP: N]` — приложение обновляет счётчик шагов
- Пользователь не видит UI-управление шагами

### Между этапами — гибридное подтверждение (кнопки + текст)

Когда LLM считает этап завершённым:

1. LLM ставит маркер `[STAGE_COMPLETE]` в конце ответа
2. Под ответом ассистента появляются **2 кнопки**: согласие / отказ
3. Если пользователь нажал кнопку → приложение автоматически вставляет текст от имени пользователя и меняет stage
4. Если пользователь проигнорировал кнопки и написал свой текст → LLM получает этот текст, видит `[STAGE_COMPLETE]` в истории и сама решает, завершён ли этап
5. Если LLM решила что этап не завершён → следующий ответ будет без `[STAGE_COMPLETE]`, кнопки пропадут
6. Если LLM снова решит что этап завершён → `[STAGE_COMPLETE]` снова появится

**Важно:** смена этапа происходит ТОЛЬКО по кнопке. Анализ текста пользователя не делается — LLM сама управляет маркером.

### При смене этапа — принудительный summary
- Берутся ВСЕ сообщения текущего этапа
- Генерируется summary: что обсудили, к чему пришли
- Вставляется system-сообщение: `📝 Завершён этап "{название}": {summary}`
- Контекст очищается от сырых сообщений этапа

### Регулярная суммаризация
- Остаётся как есть (каждые N сообщений внутри этапа)
- Порог увеличивается с 20 до 30 сообщений
- Предотвращает переполнение контекста внутри длинного этапа

### REOPEN (возврат в EXECUTION)
- LLM может предложить вернуться к выполнению если видит новый запрос
- Кнопка в UI чата "🔄 Продолжить выполнение" (или "Вернуться к подбору")
- При возврате — summary предыдущего этапа EXECUTION сохраняется в контексте

## Кнопки в UI

### Фиксированные пары кнопок (не зависят от контекста)

| Текущий этап | → Цель | Кнопка 1 (согласие) | Кнопка 2 (отказ) |
|-------------|--------|---------------------|------------------|
| PLANNING | → EXECUTION | `✅ Приступить к выполнению` | `🔄 Продолжить уточнение` |
| EXECUTION | → DONE | `✅ Завершить задачу` | `🔄 Нужно ещё доработать` |
| DONE | → REOPEN | `🔄 Вернуться к выполнению` | `❌ Нет, спасибо` |

### Текст, который вставляется в чат при нажатии кнопки (от имени пользователя)

| Нажатая кнопка | Текст сообщения |
|---------------|----------------|
| ✅ Приступить к выполнению | `→ Приступаем к выполнению` |
| 🔄 Продолжить уточнение | `→ Давай ещё уточним` |
| ✅ Завершить задачу | `→ Завершаем задачу` |
| 🔄 Нужно ещё доработать | `→ Нужно ещё поработать над этим` |
| 🔄 Вернуться к выполнению | `→ Вернёмся к выполнению` |
| ❌ Нет, спасибо | `→ Нет, спасибо` |

### Отображение
- Кнопки появляются ПОД последним сообщением ассистента
- Если ассистент не поставил `[STAGE_COMPLETE]` — кнопок нет
- Кнопки исчезают если пользователь написал свой текст (LLM получила новый запрос)

## Маркеры в ответе LLM

| Маркер | Значение | Действие приложения |
|--------|----------|---------------------|
| `[STEP: N]` | LLM перешла на шаг N | Обновить step в БД |
| `[STEPS: N]` | LLM сообщает общее количество шагов | Записать totalSteps |
| `[STAGE_COMPLETE]` | LLM считает этап завершённым | Показать кнопки под сообщением |
| `[STAGE: PLANNING]` | Целевой этап (после подтверждения) | Переключить stage в БД |

Маркеры НЕ переключают этап автоматически. Только информируют приложение и вызывают UI-кнопки.

## Что передаётся LLM в каждом запросе

```
System: [Stage: EXECUTION, Step: 2/3]
─────────────────────────────────
Summary: всё обсуждение этапа PLANNING
─────────────────────────────────
📝 Завершён этап "Планирование" (system message)
─────────────────────────────────
Summary: первые 30 сообщений этапа EXECUTION
─────────────────────────────────
User: ... (последние сырые сообщения)
```

## Изменения в хранилище (БД)

Новая таблица `chat_state`:

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | String (PK, FK→chats.id CASCADE) | ID чата |
| `stage` | String | PLANNING / EXECUTION / DONE |
| `step` | Int | Текущий шаг (0..totalSteps) |
| `totalSteps` | Int | Всего шагов на этапе |
| `updatedAt` | Long | Время последнего обновления |

---

## Режим "без этапов"

Если пользователь не использует FSM — чат остаётся в PLANNING навсегда.
Никаких предложений переключить этап. Работает как обычный чат.
